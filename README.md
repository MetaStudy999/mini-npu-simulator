# Mini NPU Simulator

반복문만으로 MAC(Multiply-Accumulate) 연산을 구현하고, `Cross`와 `X` 패턴을 판별하는 Python 콘솔 애플리케이션입니다.  
모드 1에서는 사용자가 직접 3x3 필터와 패턴을 입력할 수 있고, 모드 2에서는 `data.json`을 읽어 5x5, 13x13, 25x25 케이스를 일괄 분석합니다.

## 실행 방법

```bash
python3 main.py
```

실행 후 메뉴에서 아래 둘 중 하나를 선택합니다.

- `1`: 사용자 입력 모드
- `2`: `data.json` 분석 모드

프로젝트 루트에는 아래 파일이 포함되어 있습니다.

- `main.py`: 메인 실행 파일
- `data.json`: 필터/패턴 데이터셋
- `README.md`: 실행 방법 및 결과 리포트

## 사용자 입력 모드

1. 3x3 필터 A를 3줄 입력합니다.
2. 3x3 필터 B를 3줄 입력합니다.
3. 3x3 패턴을 3줄 입력합니다.
4. 프로그램이 A/B MAC 점수, 평균 연산 시간, 판정 결과를 출력합니다.
5. 마지막에 3x3 단일 MAC 기준 성능 표를 출력합니다.

입력 예시:

```text
0 1 0
1 1 1
0 1 0
```

입력 형식이 잘못되면 아래 안내 문구와 함께 같은 줄을 다시 입력받습니다.

```text
입력 형식 오류: 각 줄에 3개의 숫자를 공백으로 구분해 입력하세요.
```

## data.json 분석 모드

`data.json`은 아래 규칙을 따릅니다.

- `filters.size_5`, `filters.size_13`, `filters.size_25`
- 각 필터 묶음 내부 키는 `cross`, `x`
- 패턴 키는 `size_{N}_{idx}` 형식
- 각 패턴은 `input`, `expected`를 포함

프로그램은 다음 순서로 동작합니다.

1. 필터를 로드하고 스키마를 검증합니다.
2. 패턴 키에서 `N`을 추출해 대응하는 `size_N` 필터를 선택합니다.
3. `expected`의 `+`, `x` 값을 내부 표준 라벨 `Cross`, `X`로 정규화합니다.
4. Cross 필터와 X 필터 각각에 대해 MAC 점수를 계산합니다.
5. 두 점수 차이가 `1e-9` 미만이면 `UNDECIDED`로 처리합니다.
6. 판정 결과와 `expected`를 비교해 `PASS` 또는 `FAIL`을 출력합니다.
7. 마지막에 전체 테스트 요약과 성능 표를 출력합니다.

## 구현 요약

- 외부 라이브러리 없이 2차원 리스트와 중첩 반복문으로 MAC 연산을 구현했습니다.
- `PatternMatrix` 클래스에 `get`, `set` 메서드를 두어 값 저장/조회 요구사항을 충족했습니다.
- 라벨 정규화는 `+ -> Cross`, `cross -> Cross`, `x -> X` 규칙으로 통일했습니다.
- 부동소수점 비교는 `abs(score_cross - score_x) < 1e-9` 정책을 사용했습니다.
- JSON 분석 중 스키마 오류나 크기 불일치가 발생해도 프로그램 전체가 중단되지 않고 케이스 단위 `FAIL`로 처리되도록 만들었습니다.
- 성능 분석은 I/O를 제외하고 MAC 함수 호출만 10회 반복 측정해 평균 ms를 계산합니다.
- 보너스 성격으로 `Cross`, `X` 패턴 생성 함수를 추가해 성능 분석용 입력을 자동 생성합니다.

## 결과 리포트

이번 기본 데이터셋(`data.json`) 기준 실행 결과는 `총 9개 / 통과 9개 / 실패 0개`였습니다.  
실패가 0개였던 이유는 첫째, 필터 키와 expected 라벨을 프로그램 내부에서 `Cross`, `X` 두 종류로 정규화했기 때문입니다.  
만약 이 정규화가 없으면 `+`와 `cross`는 같은 의미인데도 문자열 비교에서 다른 값으로 취급되어 불필요한 FAIL이 생길 수 있습니다.  
둘째, 부동소수점 패턴(`softened_cross`, `softened_x`)을 일부 섞어 넣었지만 최종 비교는 epsilon 기반으로 처리했기 때문에 극미한 반올림 오차가 판정 오류로 이어지지 않았습니다.  
예를 들어 `22.499999999999993`처럼 표현되는 점수는 내부적으로는 기대한 값과 거의 같지만, 문자열 그대로 비교하면 오판 가능성이 있습니다.  
셋째, `patterns`의 키에서 크기를 추출하고 해당 크기의 필터만 선택하도록 해서 13x13 패턴이 25x25 필터와 비교되는 구조적 오류를 막았습니다.  
넷째, 필터나 패턴의 행/열 수가 맞지 않으면 예외로 프로그램이 죽지 않고 케이스 단위 FAIL 사유를 남기도록 설계했습니다. 즉, 데이터 문제와 로직 문제를 분리해 진단할 수 있습니다.  
시간 복잡도는 MAC 연산이 모든 좌표를 한 번씩 방문하므로 `N x N` 입력에 대해 `O(N^2)`입니다.  
실제로 연산 횟수는 3x3에서 9회, 5x5에서 25회, 13x13에서 169회, 25x25에서 625회로 증가하며, 측정 시간도 같은 방향으로 증가했습니다.  
한 번의 측정 예시에서는 3x3 약 `0.002 ms`, 5x5 약 `0.004 ms`, 13x13 약 `0.017 ms`, 25x25 약 `0.057 ms` 수준이었습니다.  
절대 시간은 실행 환경에 따라 달라질 수 있지만, 입력 크기가 커질수록 평균 연산 시간이 함께 증가하는 경향은 `O(N^2)` 분석과 일치합니다.  
정리하면 이 구현에서 중요한 안정화 포인트는 라벨 정규화, 크기 검증, epsilon 비교 정책이며, 이 세 가지가 맞물려야 데이터/스키마/수치 오차 때문에 생기는 가짜 실패를 줄일 수 있습니다.

## 재현성 체크

- 모드 1에서 3x3 `Cross` 필터와 `X` 필터를 입력하면 점수와 판정이 정상 출력됩니다.
- 모드 2에서 `data.json` 분석을 실행하면 각 케이스의 `PASS/FAIL`과 최종 총합이 일치합니다.
- 잘못된 사용자 입력은 재입력을 유도하고, JSON 스키마 문제는 케이스 단위로 보고됩니다.
